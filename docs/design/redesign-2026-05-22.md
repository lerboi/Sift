# Sift — UI/UX Redesign Brief (2026-05-22)

User audit after tick 44 surfaced four structural issues that the iteration loop's polish passes can't fix from the inside. This brief is the plan: research, options weighed, decisions made, screen-by-screen specs, and an explicit checklist the loop will execute against (see [`iteration-plan.md`](iteration-plan.md) § Phase R).

Treat this file as **the redesign's source of truth**. If the loop discovers something mid-flight that contradicts a decision here, update this file first, then proceed.

---

## The four critiques (verbatim, paraphrased)

1. **Ticker detail is unclear what you're looking at.** No organizing principle; the screen is a bag of sections.
2. **Home doesn't anchor dates.** Hard to see what's upcoming vs recent, no clear timeline, "in 2h / tomorrow / wed" copy loses the calendar.
3. **Home and Watchlist feel similar.** Both are vertical ticker lists; Home doesn't have a distinct job.
4. **Third tab (Events) doesn't justify itself.** Re-evaluate whether it should be a top-level destination at all.

Underneath all four: **the app shows information, but the user can't tell *what they're looking at* at a glance.** The fix is structural (IA + screen identities), not surface polish.

---

## Decisions, in one paragraph

1. **Drop the Events tab.** Replace with a new tab: **Discover** — global search + this-week's biggest expected surprises + sector heat. Events' filtering functionality folds into Discover as one section.
2. **Reframe Today (was Home)** as a **chronological day-grouped timeline**, with absolute dates as the primary visual anchor (`TUE · MAY 26`), and ticker as supporting metadata inside each row.
3. **Reframe Watchlist** as a **ticker-first portfolio surface** (symbol + sparkline + sector + next-earnings countdown) — not an event feed. Visually distinct from Today: rows, not cards.
4. **Re-architect ticker detail** around a **vertical timeline spine**: a hero block, then a single chronological stream of events / briefings / transcripts flowing future-to-past, every item time-stamped with an absolute date.

The full reasoning, alternatives considered, and screen specs follow.

## Research: what comparable apps do

| App | Top tabs | Pattern that matters |
| --- | --- | --- |
| **Apple Stocks** (iOS 26) | My Symbols, Business News, Top Stories | 2-of-3 tabs are content feeds; tickers live inside My Symbols. No "events" tab. |
| **Robinhood** | Investing, Spending, Crypto | Earnings live inline on each ticker's detail screen, not as a calendar. |
| **Yahoo Finance** | News, Watchlists, Markets, Search, Portfolios | "Earnings calendar" is buried inside Markets — not promoted to top-level. |
| **Seeking Alpha** (closest analog — research-focused) | Home, Portfolio, Top Stocks, News, Premium | Home is a research feed; Top Stocks is a discovery surface. No calendar tab. |
| **Webull** | Markets, Watchlists, Trade, Menu | Earnings calendar inside Markets; never promoted. |

**Pattern:** no widely-shipped consumer financial app makes "earnings calendar" a top-level tab. They put it inside a Markets/Discover surface. Sift's Events tab is an anomaly that doesn't pay rent.

**Anchor for Sift's identity:** Apple Stocks for chrome and calm; Seeking Alpha for the research feed framing; Apple Health Studies for the disclaimer model. Avoid Robinhood/Webull for visual treatment (too excited).

## Information architecture — old vs new

### Old (today)

```
Tabs:   [ Today ]  [ Watchlist ]  [ Events ]  [ Settings ]

Today:        feed of LIVE / UPCOMING / RECENT cards (your watchlist)
Watchlist:    your tickers, grouped by week
Events:       feed of all events, filterable by scope/outcome/ticker
              ← essentially "Today with no watchlist filter + chips"
Settings:     account / notifications / about
```

Problem: Today and Events render the same content type (event cards) with overlapping data. Today and Watchlist both render ticker-keyed vertical lists.

### New

```
Tabs:   [ Today ]  [ Watchlist ]  [ Discover ]  [ Settings ]

Today:        chronological timeline of YOUR events (briefings, live, past)
              date-grouped — date is the primary anchor, ticker is secondary
              event-first rendering — looks nothing like Watchlist

Watchlist:    your tickers as a portfolio surface
              ticker + sparkline + sector + next-earnings countdown
              ticker-first rendering — looks nothing like Today

Discover:     search bar (global ticker lookup)
              this-week's biggest expected surprises (cross-market)
              sector heat (which sectors are reporting heavily this week)
              browse mode that absorbs Events' "all events" feed

Settings:     unchanged
```

**Why three content tabs not two:** the FYP's value-add (the ML model) only sings when users can see *cross-market* signal, not just their watchlist. A 2-tab IA (Today + Watchlist + Settings) hides Sift's USP behind a search action. Discover surfaces it without polluting Today.

**Future evolution flagged** — Phase 12 (post-MVP) may rename Discover to Insights once the model has accumulated a track record. For now, "discover" is more honest about what the surface does pre-model-maturity.

### Alternatives considered for the third tab

| Option | Why not | Why we considered |
| --- | --- | --- |
| **Drop entirely (2 tabs + header search button)** | Hides cross-market discovery behind a chrome action; weakens the FYP's "market intelligence" framing. | Cleanest mobile shape. Apple Stocks ships this. |
| **Rename to Insights (model performance + your prediction track record)** | Has no content pre-launch — the model needs months of live data to be interesting. Empty Insights tab = ghost town. | Strongest long-term value prop for the FYP narrative. |
| **Rename to Briefings (cross-market briefing feed)** | Briefings are per-event; surfacing them outside an event's context is incoherent. | Reuses generated content. |
| **Discover (chosen)** | — | Real job today (search + cross-market signal), evolves to Insights when data justifies it. |

#### Key changes from current Home

- **Date eyebrow is the primary anchor.** Format: `TODAY · TUE MAY 26` or `WED · MAY 27` (relative + absolute, with weekday for scannability). Replaces today's `LIVE NOW / UPCOMING / RECENT` section labels.
- **Each card carries an absolute time.** `4:00 PM ET · After close`, `Pre-market`, `LIVE · filed 4m ago`. Never just "in 2h".
- **Single card shape across upcoming + live + past.** The shape changes by state (live ribbon, briefing-ready dot, completed outcome line), but the layout is consistent. Reduces cognitive load — users learn one card.
- **Live status is *neutral*, not red.** Current `signal.negative` for LIVE conflates "live" with "bad" (red is P&L colour). Use `accent.default` with a filled half-circle glyph `◐` for live, paired with the word "LIVE."
- **No more "section first" gymnastics.** Each day group is self-contained; the order is purely chronological. If today has nothing, the screen starts at "tomorrow."
- **Empty state survives:** if zero days have anything, render `EmptyState` with CTA to add tickers (unchanged from current).

#### Component changes

- New `<DayHeader date relativeLabel />` — renders `TODAY · TUE MAY 26` typography. Reusable on Discover.
- New `<EventTimelineCard state="upcoming" | "live" | "past" />` — replaces `EarningsCard` + `EventCard` from Home (those two components were 80% the same). Single source of truth for the event-as-card visual language.
- Retired: `<EarningsCard>` and `<EventCard>` (Today usage). Keep them around until Discover migration completes, then delete.

#### Data shape change (mock + future schema)

Every event needs an `expectedAt: ISO8601` and `actualAt: ISO8601 | null`. The day-grouping uses `expectedAt || actualAt`. Existing mock data's `when: "in 2h"` strings become derived from `expectedAt`.

### Watchlist

**Job:** show me the tickers I follow, with enough context to decide whether to drill in.

**Identity:** ticker symbol is the strong typography. Sparkline + sector are the supporting data. Next-earnings is one piece of metadata, not the focal point.

#### Structure

```
┌─────────────────────────────────────┐
│  Watchlist                       +  │  large title + add button
│                                     │
│  Sort: ⌄ Next earnings              │  ← sort selector (subtle)
│                                     │
│  ┌─────────────────────────────────┐│
│  │ AAPL    ╱╲╱╮       Q1 26   2d   ││  ← row
│  │ Apple Inc.            ● ready    ││
│  │ ─────────────────────────────── ││
│  │ MSFT    ╱╮╲╱       Q2 26   3d   ││
│  │ Microsoft Corp.                  ││
│  │ ─────────────────────────────── ││
│  │ NVDA    ╲╱╮╲       Q1 26   8d   ││
│  │ NVIDIA Corp.          ● ready    ││
│  │ ─────────────────────────────── ││
│  │ GOOG    ╮╲╱╲       Q1 26   9d   ││
│  │ Alphabet Inc.                    ││
│  └─────────────────────────────────┘│
│                                     │
│  AMZN, META, TSLA below the fold…   │
└─────────────────────────────────────┘
```

#### Key changes from current Watchlist

- **Drop the THIS WEEK / NEXT WEEK / LATER section bucketing.** Replace with a single sortable list. Section bucketing duplicates work the date column already does and fragments the visual rhythm.
- **Add a Sort selector** in the screen header — `Next earnings (default) · Alphabetical · Recently added`. Cheap to wire, gives the screen agency.
- **Each row is denser, single-line-driven.** Two visual rows per ticker (symbol+name+meta on row 1; sparkline+date+countdown on row 2) was confusing. New: symbol on left, sparkline mid-prominent in the middle, period+countdown right.
- **Sparkline becomes muted by default.** Per Bug log B16 — green/red sparklines on mock data imply trend we haven't computed. Default to `text.secondary` tint until real 30d-change data lands, then re-tint.
- **Briefing-ready becomes a row badge,** not a dot in the corner — `● ready` text under the name, so it's scannable as content not chrome.

#### Component changes

- Update `<WatchlistRow>` — new layout, drop the SPARKLINE_COLOR_BY_TREND map (mute by default).
- New `<SortSelector value onSelect options />` — pill-like dropdown in the header.
- Retire `groupByWeek()` from `mock.js` (keep export for one tick, delete the call site).

#### Decisions

- **Tapping a ticker row** → ticker detail (`/watchlist/<symbol>`) even if not on watchlist. CTA on that screen ("Add to watchlist") handles intent.
- **Old Events screen's filter/scope/outcome chips** are not reproduced here. They were filters over a feed the user didn't need. If we miss them later, add a "Browse all events" link at the bottom of Discover that opens a filtered sub-screen.

- Retire Events tab: remove `app/(app)/events/_layout.js`, `app/(app)/events/index.js`. Event detail (`app/(app)/events/[event_id].js`) stays — it's reached from Today and from ticker history. Move it under a non-tab group if expo-router complains, otherwise keep the events folder solely for the detail route. **TBD during the migration tick** — verify which layout pattern works without re-introducing a tab.

#### Key changes from current ticker detail

- **One chronological spine.** All items (next earnings, past events, briefings, transcripts) flow in a single date-stamped stream, not separate sections. Future at top, past going down.
- **Hero is tightened.** One line for `name · sector` (was two). Sparkline gets a `+2.1% 30d` adjacent label so the line has data, not just shape. Drop the empty space below.
- **Every history item has a date eyebrow.** Same `<DayHeader>` component as Today — keeps the visual language coherent across screens.
- **`UP NEXT` block becomes the first timeline item.** Same card shape, just a future date. Loses its bespoke "UP NEXT" label.
- **PAST BRIEFINGS / PAST EVENTS / TRANSCRIPTS sections collapse into the spine.** They were the three biggest culprits making the screen feel like a TODO list of sections. Each item is now a card with a date and a glyph (`⏰` upcoming, `✓` reported, `📄` briefing, `💬` transcript).
- **Methodology sheet** trigger stays — moves to the `ⓘ` next to "Beat probability" inside the upcoming card. Sheet content unchanged.
- **Sticky CTA bar** stays as-is — toggle add/remove from watchlist.

#### Component changes

- New `<TimelineItem date glyph kind>` wrapper — paints the day header + the card, owns the spacing rhythm. Eats `EarningsCard`, `EventCard`, `BriefingCard`, `TranscriptCard` as `children` while the migration happens; later may absorb them.
- `<BriefingCard>` and `<TranscriptCard>` keep their internal layout but get embedded under `<TimelineItem>` instead of grouped sections.
- Drop the "TRANSCRIPTS" / "PAST BRIEFINGS" / "PAST EVENTS" section labels entirely — date eyebrows are the new structure.

#### Failure mode to watch

If the timeline mixes content types at uneven cadence (e.g., one quarter has briefing+transcript+event, another has only an event), the visual rhythm could feel uneven. Mitigation: each card has a consistent vertical footprint (~120pt min), and the glyph in the top-left tells the user the content type at a glance.

## Acceptance: how we know Phase R is done

- **Today and Watchlist look unmistakably different at a glance.** Show a friend both screens for 1 second each; they can describe the difference.
- **Every event surface (Today, ticker timeline, Discover rails) carries an absolute date.** Search the codebase for `"in 2h"` / `"tomorrow"` / `"wed"` mock strings — they should be gone or derived from real timestamps.
- **The ticker detail page reads as one timeline.** No section labels other than date eyebrows.
- **The tab bar has Discover, not Events.** Tab 3 has a clear and distinct job.
- **The bundler boots clean. `expo-doctor` 17/17.** No regressions in visuals on web or native.

When all ten R-items are checked and acceptance passes, the redesign is done and the loop returns to Phase 7 (Settings) to continue forward build.

## Open uncertainties (flag during execution)

- **Date format for international users.** `MAY 26` is US-centric. The product is US-equity-only (Russell 1000), so US conventions match the audience — but verify locale handling doesn't render `26/05` on a UK device in a way that fights the day-eyebrow rhythm.
- **Live state ribbon.** The proposal uses `accent.default` for live, dropping the existing red treatment. Verify this reads as "live" and not "branded chrome" once shipped — adjust during R9 if the affordance is unclear.
- **Discover's "biggest expected surprise" rail compliance.** Surfacing predictions for non-watchlisted tickers may push closer to "recommendation" framing than the watchlist context does. Copy must stay strictly model-output language — `"Model expects ±5.2% surprise"`, never `"NVDA likely to surge"`. Compliance-review this rail's copy before shipping R7.
