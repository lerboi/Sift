# Supabase + Postgres conventions

A reference card. Skim once, internalise, then come back when you're about to break a rule.

## Schema

- **`timestamptz` everywhere**, never `timestamp`. Postgres stores UTC; clients convert. The `timestamp` (without tz) type is a footgun — it's *not* "local time", it's "time without any zone information at all", and downstream code will silently misinterpret it.
- **`uuid` for primary keys**, generated by `gen_random_uuid()` (pgcrypto, enabled by default on Supabase). Surrogate keys are immutable and don't leak business meaning.
- **`text` for strings, never `varchar(n)`**. Postgres performance is identical; the length cap on `varchar` causes painful migrations when you outgrow it. Constrain via `CHECK` if a real bound exists.
- **`jsonb` not `json`**. `jsonb` is binary-parsed at write time, supports indexing (`->`, `->>`, `@>`), and is faster on read. Sift has no streaming-JSON use case.
- **`numeric` for money / EPS / percentages**, never `float` / `double precision`. Floating-point error compounds in financial reporting.
- **Generated columns for derivations.** `eps_surprise_pct = (eps_actual - eps_est) / NULLIF(eps_est, 0)` — defined once in DDL, can't be wrong client-side. Use `STORED` for queryable/indexable derivations, `VIRTUAL` for compute-on-read (Postgres 18+; until then, view-based).
- **Foreign keys always indexed manually.** Postgres does **not** auto-index FK columns. Every `FOREIGN KEY` declaration gets a paired `CREATE INDEX`. Skipping this causes table-scan joins that get slow at thousands of rows.

## Naming

| Thing | Pattern | Example |
| --- | --- | --- |
| Table | snake_case, plural | `tickers`, `briefings`, `event_metrics` |
| Column | snake_case | `expected_release_at`, `user_id` |
| Boolean | `is_*` / `has_*` | `is_active`, `has_consensus` |
| Timestamp | `_at` suffix | `created_at`, `filed_at` |
| FK (uuid) | `<singular>_id` | `event_id`, `transcript_id` |
| FK (symbol) | `ticker_symbol` | only one of these; tickers are PK'd on symbol |
| Index | `idx_<table>_<cols>` | `idx_events_ticker_filed_desc` |
| Policy | `"<verb> <scope>"` | `"own profile select"`, `"all auth read tickers"` |
| Trigger | `<table>_<event>_<purpose>` | `events_after_insert_notify` |
| Function | `<verb>_<noun>()` | `enforce_push_throttle()`, `home_events_for_user()` |
| Enum type | `<scope>_<concept>` | `event_source`, `notification_kind` |

## RLS

- **Default-deny on every table.** `ALTER TABLE <t> ENABLE ROW LEVEL SECURITY;` in every migration that creates a table. Without it the table is publicly readable. *This is the single most common Supabase bug.*
- **One policy per (table, action) pair, narrowly scoped.** Don't write a "god policy" with complex `OR`s — they're impossible to audit.
- **`auth.uid() = user_id`** for user-owned tables. Cache this as the canonical pattern.
- **`TO authenticated USING (true)`** for catalog reads. `service_role` bypasses RLS, so no write policy is needed for catalog tables — only Modal/edge functions write.
- **Test policies in the Supabase SQL editor** with `SET ROLE authenticated; SET request.jwt.claim.sub = '<user-uuid>'`. The CLI doesn't enforce RLS in `db push`.

## Migrations

- **Use the Supabase CLI** (`supabase migration new <name>`). Never click-through the dashboard for DDL — that leaves the migration tree drifted from the live schema.
- **One concern per migration file.** A migration that creates `briefings` shouldn't also add a column to `profiles`. Smaller migrations are easier to roll back conceptually.
- **Forward-only.** Once a migration is applied to a shared environment (staging, production), it's immutable. New changes go in a new file.
- **Idempotent where safe.** `CREATE TABLE IF NOT EXISTS` is fine for initial creation; `ALTER TABLE ... ADD COLUMN IF NOT EXISTS` for adds. Avoid `IF NOT EXISTS` on constraints — silent skips hide drift.
- **DDL and DML separately.** Schema in one migration; seed data in the next. Lets you re-seed without re-running schema.
- **`supabase db reset` locally before push.** Catches order-dependence bugs.

## Indexes

Always-index list (no judgement required):

- Every `FOREIGN KEY` column
- Every column used in a `WHERE` clause of a hot query
- `(user_id, created_at DESC)` for any per-user feed
- Sort columns when paired with filter (`(status, created_at)` for "pending oldest first")

Use **partial indexes** for sparse queries:

```sql
CREATE UNIQUE INDEX idx_model_versions_one_active
  ON model_versions (kind)
  WHERE status = 'active';
```

Enforces "exactly one active row per kind" without checking every row on every query.

Use **expression indexes** for case-insensitive matches:

```sql
CREATE INDEX idx_tickers_symbol_upper ON tickers (upper(symbol));
```

(Not needed for tickers since we store uppercase. Documented as the pattern when needed elsewhere.)

## Realtime

- **Realtime inherits RLS.** A client subscribing to `notifications` filtered by their own `user_id` works because the RLS policy already filters reads to `auth.uid() = user_id`. Anything they can't `SELECT`, they can't subscribe to. Don't try to bypass.
- **Filter at the channel, not the client.** `filter: 'user_id=eq.<uuid>'` cuts traffic at the server. Filtering in JS after the row arrives is wasteful and leaks intent.
- **Always unsubscribe in cleanup.** `useEffect` returns an unsubscribe; failing to call it leaks WebSocket capacity.
- **Don't subscribe to the catalog firehose without a filter.** `events` table inserts ~2000 rows/year aggregate, but ticker-level subscriptions stay tight (~4-8 per year per ticker).

## Edge Functions (Deno)

- **One function per concern.** `notify_user_event`, `cleanup_old_notifications` — don't mux purposes.
- **Hold the `service_role` key in the function's environment**, never in the request payload.
- **Idempotency keys** for any function that can be re-invoked by a retried trigger. Use `notifications.id` or a composite (`user_id`, `event_id`) check before insert.
- **Return early with structured logs.** Edge Function logs are searchable in the dashboard; structure them as JSON.
- **Set a timeout.** Default 60s is fine for fan-out; tighten when latency-critical.

## Storage

- **Private buckets by default.** Public buckets are public to the entire internet.
- **Signed URLs for time-limited access** (e.g. download a transcript file). 60s default, max 1h for human-handoff use cases.
- **Bucket per concern.** `raw-filings`, `transcripts`, `models`, `public-assets` (only this last one is public). Mixing breaks the per-bucket ACL model.

## Connection pooling

- **Supavisor in "transaction mode"** for serverless writers (Modal jobs, Edge Functions). Connection-per-request would exhaust the pool fast.
- **Long-lived connections for in-app realtime** are fine — they hold one WebSocket, not one Postgres connection.

## Extensions to enable

| Extension | Why |
| --- | --- |
| `pgcrypto` | `gen_random_uuid()`. Usually enabled by default on Supabase. |
| `pgvector` | Transcript-segment embeddings + cosine similarity for novelty detection. |
| `moddatetime` | Auto-update `updated_at` columns via trigger. |
| `pg_cron` | Scheduled SQL jobs (notification cleanup, daily rollups). Supabase Pro+. On Free, schedule from Modal instead. |
| `pg_stat_statements` | Query performance diagnostics. Enabled on Supabase by default; use the dashboard. |

Enable in a dedicated migration (`20260525_001_extensions.sql`) so the order is explicit.

## What we explicitly don't do

- **No client-side computation of derived values.** Surprise % is server-computed; the client never multiplies. Mirrors P11-2's "primitives enforce, don't trust the caller."
- **No `SECURITY DEFINER` functions exposed to clients without RLS gating.** They bypass RLS by definition — only use them from triggers or for service-role utilities.
- **No multi-tenant joins for "browse other users' watchlists".** Sift is impersonal; users see only their own.
- **No deleting catalog rows.** `tickers.is_active = false` for delisted; events/briefings/transcripts stay forever. Storage is cheap; audit trail matters more.
- **No application-level sequences for IDs.** `gen_random_uuid()` everywhere. Auto-increment integers leak business metrics (row counts) and complicate distributed writes.
